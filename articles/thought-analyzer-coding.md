AIに「いい感じにして」と頼んで、期待と違う結果が返ってきたことはあるだろうか。問題はAIの性能ではなく、指示の構造にある可能性が高い。thought-analyzerの新しいskillは、この「指示の質」を6つの軸で可視化する。

## 「コードが書けない」は関係ない ── 問われるのは指示の構造だ

AIとの協働が日常になる中で、一つの問いが浮かぶ。コードを書けなくてもAIに指示できれば、それは「技術力」と言えるか。

部分的にはそうだ。だが指示の質は均一ではない。曖昧な要件を投げ続ける人と、境界条件まで構造化して伝えられる人では、同じAIを使っても結果が変わる。

これは技術の問題というより、認知の問題だ。Jeannette Wing（2006）が提唱した「Computational Thinking」──分解・抽象化・パターン認識──は、AIへの指示にそのまま適用できる。

## 6軸で何を測るか

```
┌──────────────────┬────────────────────────────────────────┐
│  何を伝えるか     │  specification_precision               │
│                  │  → 要件を分解・精密化する力              │
├──────────────────┼────────────────────────────────────────┤
│  何を見るか       │  error_recognition                     │
│                  │  → 動作ではなく構造で問題を掴む力         │
├──────────────────┼────────────────────────────────────────┤
│  何を把握するか   │  system_abstraction                    │
│                  │  → コンポーネント間の関係を読む力         │
├──────────────────┼────────────────────────────────────────┤
│  何を判断するか   │  decision_quality                      │
│                  │  → 新しい状況で判断の枠組みを変える力     │
├──────────────────┼────────────────────────────────────────┤
│  何を使うか       │  technical_vocabulary                  │
│                  │  → 専門用語を正確な文脈で使う力           │
├──────────────────┼────────────────────────────────────────┤
│  どう回すか       │  iteration_style                       │
│                  │  → フィードバックの粒度と明確さ           │
└──────────────────┴────────────────────────────────────────┘
```

### ① specification_precision（要件定義の精度）
*理論：Computational Thinking — Wing（2006）*

「何を作りたいか」を分解して正確に伝えられるか。スコア1〜5で測定する。

```
1 ── 目的・制約が曖昧。「いい感じにして」
2 ── 目的はあるが制約・境界条件が不明
3 ── 目的と主要な制約を伝えられる
4 ── 目的・制約・優先順位を構造的に伝えられる
5 ── エッジケース・非機能要件まで含めて精密に定義できる
```

### ② error_recognition（エラー・誤りの認識力）
*理論：Expert-Novice distinction — Chi, Feltovich & Glaser（1981）*

専門家と初学者の違いは「何を見て問題を認識するか」に現れる。表面的な動作で判断するか、構造的なパターンで判断するかの差だ。

```
structural    ── エラーの構造・原因を推定して指摘できる
behavioral    ── 動作が期待と違うことは分かるが原因は不明
result_only   ── 最終結果の正否のみで判断する
```

### ③ system_abstraction（システム思考の抽象度）
*理論：Computational Thinking — Abstraction（Wing, 2006）*

コンポーネント間の関係・依存関係をどのレベルで把握して指示しているか。

```
architecture ── システム全体の構成・依存関係を把握して指示する
component    ── 個別コンポーネントの役割を理解して指示する
interface    ── 入出力・操作レベルで理解している
blackbox     ── 中身を問わず結果のみで判断する
```

### ④ decision_quality（技術的判断の質）
*理論：Adaptive Expertise — Hatano & Inagaki（1986）*

ツール選択・ライブラリ選定・ピボットのタイミングを何に基づいて判断しているか。既知のパターンを適用するだけか（Routine Expertise）、新しい状況で判断の枠組み自体を変えられるか（Adaptive Expertise）の差だ。

```
adaptive  ── 状況に応じて判断の枠組み自体を変えられる
routine   ── 既知のパターンを適用する
deferred  ── 判断を相手（AI・エンジニア）に委ねる
```

### ⑤ technical_vocabulary（技術語彙の正確性）
*理論：Domain-specific knowledge transfer — Ericsson（1993）*

専門用語を正確に・適切な文脈で使えるかどうか。誤用・過少使用・過剰使用の三方向で評価する。

```
precise      ── 専門用語を正確な意味で使う
approximate  ── 概念は正しいが表現が近似的
lay          ── 日常語で説明する（正確さより分かりやすさ優先）
mixed        ── 文脈によって使い分けている
```

### ⑥ iteration_style（改善サイクルの回し方）
*理論：Self-Regulated Learning — Zimmerman（2000）*

フィードバックを与えて改善していく際のパターン。粒度（小刻みか一括か）と明確さ（クリアか曖昧か）の2軸で分類する。

```
incremental_clear ── 小さく・明確なフィードバックを積み重ねる
batch_clear       ── まとめて・明確なフィードバックを一度に出す
incremental_vague ── 小刻みだが指示が曖昧になりやすい
batch_vague       ── まとめて出すが優先順位が不明確
```

## マークダウン構造は「補助」でしかない ── 形式と中身を混同してはいけない

Claudeを使いこなすユーザーの多くは、指示をマークダウンで書く。`##` でセクションを切り、コードブロックで期待する入出力を示す。この書き方には思考構造が滲み出る。

だが、構造化されたマークダウンが高精度な要件定義を意味するとは限らない。

```
【形式だけ】                        【形式＋中身】

## 要件                            ## 要件
- いい感じにして                    - ボタンクリックで一覧を絞り込む
                                   - 検索は部分一致、大文字小文字を無視する
## 制約                            ## 制約
- 特になし                          - レスポンスは200ms以内
                                   - モバイルでも動作すること

→ specification_precision: 1       → specification_precision: 4
```

形式だけ整って中身が空なケースだ。この状態で「specification_precisionが高い」と判断すれば誤診になる。さらに、マークダウン構造を正確に診断するには複数セッションの一貫性が必要で、一度だけの使用では習慣か偶発かを判断できない。

このためマークダウンの書き方は独立した軸とせず、6軸の補助情報として `commentary` に反映している。形式と中身が一致するときだけ有意なシグナルとして読む。

## 思考パターン分析との違い ── 2つのskillは何が見えるか

```
┌──────────────────────────┬──────────────────────┬──────┐
│ skill                    │ 何を測るか            │ 軸数 │
├──────────────────────────┼──────────────────────┼──────┤
│ 思考パターン分析          │ 認知スタイル全般      │  9軸 │
│ コーディング指示力分析    │ 技術的指示・判断・制御 │  6軸 │
└──────────────────────────┴──────────────────────┘──────┘
```

思考パターン分析が「その人がどう考えるか」の地形を測るなら、コーディング指示力分析は「どうAIに働きかけるか」の癖を測る。

2つを組み合わせると傾向の一貫性が見えてくる。思考パターンで `problem_style: deferred`（判断を委ねる傾向）が出て、コーディング指示力でも `decision_quality: deferred` が出るなら、それは偶然ではなくスタイルだ。逆に両者が一致しないケースは、文脈依存の切り替えが起きている可能性を示す。

## 使い方

Claude Codeで以下を発言すると起動する。

- 思考パターン分析：「思考パターンを分析して」
- コーディング指示力分析：「コーディング指示力を分析して」

分析結果（軸の値のみ）は匿名で蓄積される。会話の内容・コード・固有名詞は一切送信されない。送信は毎回任意だ。n数が積み重なるにつれて、指示スタイルの集合的なパターンが見えてくる。
