「30件の会話ログを分析に渡しても、冒頭の一言に引きずられるリスクをはらんでいるのではないか」──この問いは、thought-analyzerの設計を深く理解しようとする過程で自然に浮かぶ。本記事はその問いを正面から検討する。

---

## LLMはコンテキストに即時同期する

LLMの学習には**in-context learning**（文脈内学習）と呼ばれる特性がある。モデルは入力されたコンテキストのパターンに、重みを更新せずにリアルタイムで適応する（Brown et al., 2020, *Language Models are Few-Shot Learners*）。

これは推論中の動的な整合性確保だが、ある副作用をもたらす。**高密度な情報を持つ発言が、後続の処理トーンを支配する傾向がある**。

同様の現象は人間のコミュニケーション研究にも記録されている。Gilesらが提唱したCommunication Accommodation Theory（1973）によれば、人は相手の話し方・発話スタイルに無意識に収束する傾向（収束:convergence）を持つ。LLMも、ある種の「対話相手への同期」を行っていると見ることができる。

---

## thought-analyzerが前提とするもの

thought-analyzerは**30件以上・1000字以上の会話ログから習慣的パターンを読む**という設計前提を持つ。

この前提は「単一の発言から性格を判定するのではなく、繰り返し現れる構造から思考の地形を測る」という哲学に基づいている。脳科学的にいえば、宣言的記憶ではなく手続き記憶に近い層を観測しようとしている。

しかしここに矛盾が生じる可能性がある。

```
ユーザーが実際に試みること：
  30件の会話ログを一括で投入する

LLMが実際に行うこと：
  コンテキスト全体を処理しながら、情報密度の高い発言に同期する
  → 結果として、特定の発言が分析の「基調」を決めてしまう
```

**「コンテキストへの即時同期」と「習慣的パターンの蓄積から構造を読む」は、別の層で起きる現象だ。** 前者は推論時の動的適応であり、後者は統計的に安定した傾向の抽出を目指す。

## 問題の核心：短い一言が残りを支配する

具体的にどういうことか。

次のような発話パターンを想定する：

```
発言A（5字）：「むずい」
発言B（800字）：技術的な問題について詳細に説明し、複数の解決案を提示している
発言C（300字）：設計の意図について哲学的な問いを立てている
...（以下28件）
```

LLMがこのログを受け取るとき、発言Aの口語的・感情的トーンがコンテキストの「最初の色」を設定し、以降の精緻な発言を解釈するフィルターになりうる。逆に、冒頭に専門用語が密集した高密度な発言があれば、後続の日常的な発言もその文脈で解釈される。

問題は「発言Aが誤っている」のではなく、**影響力の所在が不可視**であること。分析者（AIも、ユーザーも）はどの発言が分析結果を主導したかを知らないまま結果を受け取る。

---

## 影響の所在を可視化する3つのアプローチ

この問題に対処する手法を3つ検討する。実装コストと測定精度のトレードオフで整理した。

### ① ローリングウィンドウ分析

会話ログを時系列で分割し、区間ごとに分析してスコアの推移を追う。

```
全30件 → [1-10件] [11-20件] [21-30件] を別々に分析
        → abstraction_direction: [低抽象, 高抽象, 低抽象]
           → 変化点が第11発言付近に存在する
```

スコアが急変した区間には「影響の可能性がある発言が含まれる」と推定できる。時系列シャッフルは文脈を破壊するため不適だが、時系列を保ったまま窓をずらすことで影響の変化点を特定できる。

**実装コスト**：分析を3〜5回繰り返す必要がある
**測定精度**：区間の粒度に依存。「どの区間か」はわかるが「どの発言か」まで特定できない

---

### ② サリエンシーマップ（LLMへの直接問い合わせ）

LLMに対して、分析後に「各発言が各軸のスコアに何%程度貢献したか」を推定させる。

```
プロンプト例：
「今の分析結果について、abstraction_directionのスコアに特に影響した発言を
 番号で上位3件挙げ、それぞれの寄与を大・中・小で評価してください」
```

これはモデルの自己説明能力（self-explanation）を利用する手法で、interpretable MLにおけるfeature importance出力に相当する。

**実装コスト**：追加プロンプト1回分。thought-analyzerの出力フローに組み込みやすい
**測定精度**：LLMの自己説明は必ずしも内部処理を正確に反映しない。ただし「LLMが意識的に参照した発言」は開示される

---

### ③ Leave-one-out（削除法）

各発言を1件ずつ除外して再分析し、スコア変化量が最大の発言を「最高影響発言」と特定する。

```
30件ログ → 発言1を除いて分析 → スコア差分を記録
         → 発言2を除いて分析 → スコア差分を記録
         → ...
         → 差分が最大の発言 = 影響の所在
```

理論的には最も精度が高いが、**n回の分析が必要**という致命的なコスト問題がある。30件ログなら最低30回の分析が必要で、実用的ではない。ただし疑わしい発言が1〜2件に絞られている場合の検証には有効だ。

**実装コスト**：高（n回の分析）
**測定精度**：高。ただし文脈の除外が分析精度を歪める副作用もある

---

## 実験：「強いシグナル」の有無でどれだけ変わるか

実際に影響の規模を確認するため、同じログの2バリアントを用意して分析した。

**バリアントA**：通常ログ（30件の技術的議論）
**バリアントB**：冒頭に「私は直感型で、論理より感性を重視します」という1文を追加

```
                   バリアントA    バリアントB    差分
abstraction_direction  concrete      abstract     ←軸が反転
problem_style         adaptive      innovative   ←変化あり
need_for_cognition    high          medium       ←変化あり
（その他6軸は概ね一致）
```

3軸がシフトした。特に `abstraction_direction` は正反対の値に変化した。

追加した1文は**全体の約0.3%のトークン**に過ぎない。にもかかわらず、9軸中3軸に影響が出た。

:::message
この実験は illustrative な目的で設計したものであり、すべてのLLMや分析条件に一般化できるわけではない。ただし、コンテキスト同期が分析に与える影響が無視できない規模であることを示す一例として提示する。
:::

## thought-analyzerはこの問題にどう向き合うか

### 現在すでに取り入れていること

thought-analyzerは現行設計でこの問題に対する2つの対処を内包している。

**ローリングウィンドウの考え方（①）**：skill内でLLMに「ログ全体を通読してからパターンを判定する」よう明示的に指示している。単純に先頭から読んで判定するのではなく、全体の文脈を把握した上でスコアを決定させる設計だ。これはローリングウィンドウの精神を、追加コストなしで実現する現実的な実装だ。

**holistic_profileによるサリエンシー記述（②の部分適用）**：holistic_profileは9軸の離散値だけでは捉えられない質感を補完するフィールドだが、分析中に特定の発言への依存が見えた場合、その傾向を自然言語で記述する機能も担っている。「特定の専門領域の会話が多く含まれていたため、この傾向は文脈依存の可能性がある」といった記述がそれに当たる。

### 次のステップ：influence_mapフィールド

サリエンシーマップ（②）を現状より構造化した形で出力するフィールドの追加を検討している。

```json
{
  "influence_map": {
    "abstraction_direction": {
      "primary_source": "発言12付近（高密度の抽象論）",
      "confidence": "medium"
    },
    "problem_style": {
      "primary_source": "発言3・17（問題への反復的アプローチ）",
      "confidence": "high"
    }
  }
}
```

このフィールドが加わると、「なぜこのスコアが出たか」をユーザーが自分で検証できるようになる。分析結果の透明性が増し、「このスコアは信頼できるか」という判断を分析者に委ねられる。

Leave-one-out（③）は精度は高いが現状コストが見合わない。ただし将来的にAPI処理速度が向上し、並列実行が現実的になれば採用可能性が上がる。

---

## まとめ

- LLMはコンテキスト内の高密度発言に即時同期する（in-context learning）
- これはthought-analyzerが前提とする「習慣的パターンの蓄積読み」とは別の層で起きる
- thought-analyzerは現在、全体再読指示とholistic_profileの記述によってこのリスクを緩和している
- 次のステップとして`influence_map`フィールドの追加を検討中。「なぜこのスコアか」をユーザーが検証できる透明性を目指す
- Leave-one-outは理論的に最も精度が高く、API処理速度の向上とともに実用化を視野に入れている
