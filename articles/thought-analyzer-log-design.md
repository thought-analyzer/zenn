セッションをまたいだとき、会話がわずかに「リセット」された感覚を持ったことはないか。

それは気のせいではない。Claude Codeはコンテキストを圧縮し、会話履歴を `<summary>` に置き換える。その瞬間、thought-analyzerが読むべきものが消えている。

そして同じ問題が、1ターンの中にも潜んでいる。「間違えた」と「やっぱり違う」は、見た目が似ている——どちらも直前の発言を否定する短い一言だ。しかしこの2つは、思考パターン分析において**正反対の意味**を持つ。前者はノイズ、後者は最も診断価値が高い発言のひとつだ。

この区別を誰も定義していない。「何が1つの思考単位か」という問いは、セッションレベルでもターンレベルでも、まだ未解決のままだ。

---

## 現状のskillが前提としていること

thought-analyzerのStep 1はシンプルだ。

> 入力されたログからユーザーの発言のみを抽出する。AIの返答・ツール出力・コードブロック・URLは無視する。

これは `Human:` / `Assistant:` という明確な構造を前提にしている。

```
Human: Aをやって
Assistant: AをやりましたのでBも確認してください
Human: ありがとう、次にCをお願いします
```

この構造が壊れると、2つの問題が起きる。

**問題1：単体分析の汚染**
Claude Codeがコンテキスト圧縮を実行すると、会話履歴は `<summary>` ブロックに置き換えられる。このブロック内では、ユーザーの発言とAIの出力が混在した状態になる。「ユーザー発言のみを抽出する」が機能しなくなる。

**問題2：ペア分析の崩壊**
ペア分析は `(AI返答 → ユーザー反応)` のペアを単位として動く。圧縮が発生した瞬間、このペア構造が失われる。`reaction_distribution` の算出が不可能になる。

---

## 既存ツールはこの問題を解いているか

ECC（Everything Claude Code）とClaude Memの両方を調査した結果、どちらも**Human vs AIの帰属を圧縮後に明示的に保持するメカニズムを公開していない**。

ECCは圧縮前の状態を保存する `pre-compact` フックを持つが、圧縮後の帰属追跡は設計外だ。Claude MemはAIのツール呼び出しを「観察」として記録する設計で、そもそもユーザー発言の保存を主目的としていない。

この問題はthought-analyzerが独自に対処する必要がある。

---

## 「1つの思考単位」を定義する

この問いに答えるために、まず発言の種類を整理する必要がある。

### ターンの定義

**1ターン = 1往復（Human発言群 + それに対するAssistant応答）**

Assistantの返答があった時点でターンが確定する。Humanが複数発言した場合はすべて「先行指示」として1ターンに束ねる。

```
Human: Aをやって       ← ターン1の先行指示
Human: あ、Bも追加で   ← 同じターン1の先行指示
Assistant: AとBをやりました  ← ターン1の確定
Human: Cもお願いします  ← ターン2の先行指示
```

### turn_quality フラグ

問題は、Humanの発言が「本当の思考」を表しているとは限らない点だ。4種類を区別する。

| 値 | 定義 | 診断対象 |
|---|---|---|
| `normal` | 通常の発言 | ✅ |
| `addendum` | 追加要求（後から足した） | ✅ 有意義なデータ |
| `pivot` | 意図的な方向転換（「まって」「やっぱり違う」） | ✅ 最も診断価値が高い |
| `retraction` | 技術的ミス（誤操作・言い直し） | ❌ 除外 |

#### `addendum`の診断的意味

「後から追加する」こと自体がパターンを示す。`specification_precision`（要件定義の精度）の低さ、あるいは思考が発言しながら展開するスタイルの現れだ。除外するより含めた方が正確なプロファイルが得られる。

#### `pivot`の診断的意味

「まって」「やっぱり違う」はペア分析の `reject` に直接マップされる。さらに `correction_precision`（修正の粒度）の判定にも使われる。これを除外すると、ユーザーがAIと実際にどう折衝しているかが見えなくなる。

#### `retraction`の判定ロジック

技術的ミスは診断にノイズを加える。以下のヒューリスティックで自動判定する。

```
直前メッセージ ≤ 5文字
  AND 現在のメッセージが以下を含む
    → 日本語：「間違えた」「送信ミス」「無視して」「なかったことに」
    → English：「oops」「ignore that」「wrong message」「disregard」「typo」
    → is_retraction: true → 除外
```

---

## 言語をまたいだ区別の難しさ

日本語では「間違えた」と「やっぱり」は明確に区別できる。しかし英語には曖昧ゾーンが存在する。

| 表現 | 対応する思考 | 問題 |
|---|---|---|
| `oops` | 技術的ミス（retraction） | ほぼ確定 |
| `wait` / `actually` | 認知的転換（pivot） | ほぼ確定 |
| `never mind` | どちらにもなる | 文脈依存 |
| `nvm` | どちらにもなる（カジュアル） | 文脈依存 |
| `scratch that` | どちらにもなる | 文脈依存 |

**設計方針：迷ったら含める。**

除外よりも `low_confidence` フラグを付けて含める方が、分析として安全側だ。「間違えた」が1件混入してもノイズの範囲だが、「やっぱり違う」を1件除外すると診断の精度が落ちる。曖昧な場合は `turn_quality: ambiguous` → 含める・低信頼で扱う。

## セッション圧縮への対処：JOSNLログ設計

根本的な解決策は**圧縮が起きる前にログを保存する**ことだ。

### 設計の概要

```
起動時：スコープ選択（30ターン / 1日 / 1週間 / 手動）
 ↓
各ターン確定後：JSONLに1行追記
 ↓
選択スコープの80%に達したら警告
 ↓
分析実行：JSONLを読み込み → turn_quality別に処理 → 分析
 ↓
分析後：ファイル削除（確認 or 自動）
```

### JOSNLの1行構造

```json
{
  "role": "human",
  "turn_index": 12,
  "timestamp": "2026-04-05T21:30:00Z",
  "session_id": "abc123",
  "content": "やっぱり違う、Bのアプローチにして",
  "turn_quality": "pivot",
  "is_compressed": false,
  "is_excluded": false
}
```

`is_compressed: true` のターンは分析から除外し、その件数を `low_confidence` の判断材料にする。

### 保存場所と削除設計

保存先：`skills/thought-analyzer/logs/`（`.gitignore`必須）

ファイル命名：`ta-log_[セッションID]_[タイムスタンプ].jsonl`

削除ルール：
- 分析完了後に自動削除
- 次回起動時に現セッションIDと一致しないファイルを自動削除（中断されたセッションの残骸）
- TTL：24時間超のファイルは起動時に自動削除

容量の目安：30ターン ≈ 30〜100KB、1週間フル ≈ 1MB以下。

## 「何を1つの思考単位とするか」という問いの意味

この設計判断は、単なる実装の話ではない。

思考パターン分析において、「1発言 = 1思考」とは限らない。一つの問いを組み立てながら複数回送信する人もいれば、完全に構成してから一度だけ送る人もいる。`addendum` パターンが頻出するユーザーは、思考を「発言しながら展開する」スタイルを持つ可能性が高い。

逆に、`pivot` パターンが多いユーザーは「まず動かしてみて、合わなければ変える」という`problem_style: pivot`のコーディングスタイルと一致することがある。

発言の単位を正確に定義することで、はじめて「1ターンの中でどう思考が展開したか」が見えてくる。
