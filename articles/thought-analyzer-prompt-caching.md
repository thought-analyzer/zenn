DS・ML・LLM領域のチュートリアルを発信し続けているAvi Chawla（@_avichawla、Daily Dose of DS共同創業者、元Mastercard AIエンジニア）が、Prompt Cachingの解説記事を公開した。その中でこう書いている。

> "A system prompt with 20,000 tokens running over 50 turns means 1 million tokens of redundant computation billed at full price, producing zero new value."

20,000トークンのシステムプロンプトが50ターン走ると、100万トークンの冗長処理が発生する。内容は変わっていない。ただ読み直されているだけだ。

Prompt Caching自体はLLMを使っている開発者には広く知られた話だ。ただ、このツイートを読んでから改めて仕組みを掘り下げると、技術の話が単なるコスト最適化を超えて、「AIと対話する人間の側」にも同じ問いを投げかけていることに気づいた。本記事はその掘り下げの記録だ。

---

## これはLLM全般の話か、Anthropic固有の話か

**仕組みはLLM全般、実装はモデルごとに異なる。**

KV Cache（Key-Valueキャッシュ：後述する計算中間データの保存庫）自体はトランスフォーマーアーキテクチャが持つ機構で、GPT・Gemini・Claudeを問わず存在する。「同じプレフィックスを再計算しない」という原理はモデル共通だ。

ただしPrompt Cachingを明示的なAPIとして提供しているのは現時点ではAnthropicが先行している。本記事の数字（90%割引・92%ヒット率）と実装詳細（`cache_creation_input_tokens` などのフィールド）はAnthropicのAPI固有の話だ。OpenAIにも類似の機能があるが、挙動・料金体系は異なる。

**アーキテクチャの設計規律（静的を上に、動的を下に）はどのモデルでも有効だ。**

---

## KV Cacheの仕組み

LLMの推論には2つのフェーズがある。

**Prefillフェーズ**（入力一括処理）はプロンプト全体を処理する。すべてのトークン（LLMが扱うテキストの最小単位。日本語1〜2文字程度）に対して行列演算を走らせ、モデルの内部表現を構築する。計算量はトークン数の二乗に比例する（O(n²)：トークンが2倍になると処理量は4倍になる）。

**Decodeフェーズ**（逐次生成）は1トークンずつ生成する。直前の状態を参照して次を予測する。こちらはメモリ律速で、Prefillより軽い。

Prefillフェーズでは各トークンに対してQuery・Key・Valueの3ベクトルが計算される（Transformerの注意機構で使われる内部値。「このトークンがどのトークンに注目するか」を決める計算の産物）。KeyとValueは、そのトークンより前のトークン列だけで決まる。一度計算したら、同じプレフィックスが続く限り変わることはない。

**KV Cacheはこのテンソル（多次元の数値配列）を保持する。** トークン列の暗号ハッシュ（データの「指紋」。1ビット変わると全く別の値になる）でインデックス化し、同じプレフィックスのリクエストが来たときに再計算をスキップする。計算量がO(n²)からO(n)に落ちる（トークンが2倍になっても処理量は2倍で済む）。

---

## 静的部分と動的部分の分割

すべてのリクエストは2つに分けられる。

| 分類 | 内容 |
|------|------|
| 静的プレフィックス | システム指示・ツール定義・プロジェクトコンテキスト |
| 動的サフィックス | ユーザーメッセージ・応答・ツール出力 |

静的部分はキャッシュに書いて以降は読む。動的部分だけが毎回新たに処理される。Anthropicの場合、cache readはinputの10分の1の料金だ。

この分割は設計の問題だ。「静的なものを上に、動的なものを下に」という順序を守るだけで、キャッシュは機能する。

---

## ハッシュの脆弱性

ここが最も見落とされやすい。

KV Cacheはトークン列の先頭からのフルハッシュ（プロンプト全体を1つの指紋として照合する）で一致判定する。**1トークンでも変わればキャッシュ全体がミスになる。**「1+2=3」と「2+1=3」は数学的に同じだが、キャッシュとしては別物だ。

プロダクションで実際にキャッシュを壊した例をChawlaは3つ挙げている。

- システムプロンプトにタイムスタンプを注入していた
- JSONシリアライザがツールスキーマのキーを毎回異なる順序でソートしていた
- セッション中にAgentToolのパラメータを更新した

これらから導かれるルールは3つだ。

1. **セッション中にツールを変更しない** ── ツール定義は静的プレフィックスの一部
2. **セッション中にモデルを切り替えない** ── キャッシュはモデル固有
3. **プレフィックスを編集してstateを管理しない** ── 代わりに次のユーザーメッセージにreminderを追記する

---

## Claude Codeの実績

Claude Codeはこの設計規律を徹底している。

セッション開始時に20,000トークン超のシステムプロンプト・ツール定義・CLAUDE.mdを一度だけ処理する。以降のターンはこの静的部分をキャッシュから読む。会話が長くなるほどキャッシュヒット率が上がり、コストが下がる構造だ。

30分のセッションを `/cost` で確認した結果：

- キャッシュヒット率：**92%**
- コスト削減：**81%**（$6.00 → $1.15）

Prompt Cachingは追加の最適化ではない。設計として組み込まれた規律だ。

:::message
**モニタリング指標（Anthropic API）**

- `cache_creation_input_tokens`：キャッシュに書いたトークン
- `cache_read_input_tokens`：キャッシュから読んだトークン
- `input_tokens`：キャッシュなしで処理したトークン

Cache efficiency = `cache_read / (cache_read + cache_creation)` で計算できる。
:::

## 同じ構造が、人間の対話にもある

ここから先が、このツイートを読んで掘り下げたかった本題だ。

**AIと対話する人間の側にも、「静的プレフィックス」と「動的サフィックス」がある。**

エージェントのシステムプロンプトに相当するのは、会話の中で最初に確立される「概念のフレーム」だ。目的、制約、前提——これらを一度置いたら変えずに積み上げていく人と、毎ターン少しずつ言い換えながら再定義する人がいる。後者はAIの側から見ると、文脈がターンをまたいでズレ続ける。ハッシュが安定しない状態だ。

thought-analyzerは、AIとの対話ログから思考パターンを分析するツールだ（[3層フレームワーク](https://zenn.dev/thought_analyzer/articles/thought-analyzer-framework) / [9軸](https://zenn.dev/thought_analyzer/articles/thought-analyzer-9axes-theory) / [6軸](https://zenn.dev/thought_analyzer/articles/thought-analyzer-coding-6axes)）。最新の設計では**AI出力への反応**も測定対象に加えている（[第6回：ペア分析](https://zenn.dev/thought_analyzer/articles/thought-analyzer-pair-analysis)）。

対話の1ループはこうだ：

```
ユーザー → [9軸 + 6軸で測る] → AI → [reaction_patternsで測る] → ユーザー → ...
```

AIが返したものに対してユーザーがどう反応するかは4種類に分類できる：`adopt`（採用）、`modify`（修正）、`reject`（却下）、`ignore`（無視）。このうちPrompt Cachingと最も直接的に対応するのが `correction_precision`（修正の粒度）だ。

| correction_precision | 修正の様式 | Prompt Cachingで言うと |
|---|---|---|
| `surgical` | 特定の語・文・ロジックだけ置き換える | 動的サフィックスへの最小追記。静的プレフィックスは維持 |
| `directional` | 「もっとこの方向で」という大まかな再指示 | プレフィックスの一部が変わる。ハッシュが部分的にズレる |
| `holistic` | 出力全体を破棄してゼロから構築し直す | キャッシュ全破棄と同じ。静的部分ごと再設計 |

### ログで確認する

筆者が別のアプリ開発で行ったセッション（約10時間・17ターン）のログを用いて確認した。アプリの内容は伏せるが、AI開発ツールへの機能追加・修正を継続的に依頼するセッションだ。

`correction_precision`の分布は `directional`（方向修正）が中心で、`holistic`（全体再構築）は**0回**だった。一度「編集局」というアナロジーでUIの流れを定義すると、その後のターンも同じフレームの上に要素を追加し続け、フレーム自体を壊すことはなかった。「有料機能の設計は今は決めない」という判断も、プレフィックスを意図的に保留する動きだ。

`reaction_distribution`は `adopt` 47%・`modify` 29%・`reject` 12%・`ignore` 12%。採用率が高く、修正は方向修正が中心。`holistic`な全体破棄はゼロだった。

Prompt Cachingの観点で言えば、このセッションは**静的プレフィックスが安定し続けた**状態に対応する。フレームを壊さず積み上げるスタイルが、キャッシュヒット型の対話構造と重なっている。

もうひとつ、thought-analyzerが取り組んでいる「コンテキストへの即時同期」問題も関係する。LLMはコンテキストの冒頭に置かれたトーンに後続の処理が引きずられやすい——これはKV Cacheが「静的プレフィックスを支配的なシグナルとして保持する」仕組みと同根だ。最初に何を置くかが、その後の全解釈を決める。

:::message alert
このログ分析は1セッション・1ユーザーの観察だ。`correction_precision`とAPIのキャッシュ効率の定量的な相関検証は今後の課題として残る。
:::

## 結局、何が言いたいか

Prompt Cachingの本質は「何を変えないかを設計すること」だ。

LLMのインフラとして見れば、それはシステムプロンプトをセッション中に動かさないことであり、ツール定義を安定させることだ。Claude Codeはこれを徹底して92%のヒット率を実現している。

人間の思考パターンとして見れば、それは「一度確立したフレームをそのまま保ち、その上に積み上げていく」という対話スタイルだ。毎ターン前提を言い直す人は、LLMで言えばキャッシュを常にミスさせているエージェントに近い。

AIとの協働を設計として捉えるなら、問うべきは「何をうまく伝えるか」だけではない。**「何を変えないか」を意識して対話を構造化しているか**、が問われる。

thought-analyzerがAIとの対話ログを分析する目的のひとつは、この「変えないもの」と「変えるもの」の扱い方のパターンを可視化することだ。
