「自己進化するAIシステムを作った」

今週、そういう投稿が2種類流れてきた。一方は成功し、もう一方はエラーの蓄積で崩壊し始めた。どちらも「自己進化」を目指していた。なのに、なぜ結果が真逆になったのか。

## LLM wikiを「自己進化させた」とき何が起きるか

Karpathyさんのワークフロー（`raw/`ディレクトリにソースを入れ、LLMがwikiとして自動整理・更新する）に触発されて、フロントエンドとCRMのコードベースをそのシステムに接続し、自動更新するダッシュボードを作った——という試みがXで話題になった。

コンセプトは理解できる。コードをLLMに「食わせ」、LLMがwikiを更新し続け、wikiがコードの改善提案を生む。自己参照するループの完成だ。

ところがコメント欄にこんな指摘が並んだ。

> 自己参照型コードベースのリスクは、LLMが「改善」だと考えるものを最適化してしまうことであり、実際にリリースされるものではないということです。Karpathyさんのwikiコンセプトはしっかりしていますが、そこに自分の生成したコードをフィードすると、エラーが急速に蓄積してしまいます。

> 数回の更新サイクル後も正確さを保てますか？私が試した自己進化型セットアップはすべて、最終的には正しいものを自信を持って書き換え始めました。

「自信を持って正しいものを書き換える」。これが問題の核心だ。

## AlphaEvolveが解いた問題

同じ週、Google DeepMindが全く異なる「自己進化」を発表した。AlphaEvolve——AIアルゴリズムを書くAIだ。単にハイパーパラメータを微調整するのではなく、アルゴリズム自体を書き換えて自律的に進化し続ける設計で、ベンチマークで人間の科学者を上回るアルゴリズムを生成したという。

AlphaEvolveは成功している。LLM wikiの自己進化は失敗に向かう。この差はどこから来るか。

構造を比べると分岐点が見えてくる。

| | LLM wiki自己進化 | AlphaEvolve |
|---|---|---|
| 生成 | LLMがコードを更新する | LLMがアルゴリズムを書く |
| 評価 | LLMが「改善かどうか」を判断 | 数学的証明・ベンチマークスコアが評価 |
| 評価器の独立性 | **生成器と同一** | **生成器から独立** |

AlphaEvolveには評価器がある。それは数学的に正しいかどうか、ベンチマークスコアが上がったかどうか——LLMの「改善したと思う」とは独立した外部の検証だ。

LLM wikiの自己進化には評価器がない。「これは改善か？」をLLM自身が判断する。LLMは一貫して「これは良い変更だ」と答えるし、その答えに確信を持つ。結果、正しいものが書き換えられ続ける。

## 「自己進化の条件」とは何か

自己進化が機能するための条件を一言で言えば、**評価器が生成器から独立していること**だ。

- 数学的証明（AlphaEvolve）
- テスト通過率・パフォーマンス指標（CI/CDパイプライン）
- ユーザーの実際の行動データ（A/Bテスト）

これらはすべて「LLMが改善したと思うかどうか」に関係なく判定する。外部の客観的な指標だ。

逆に、評価器が生成器と同じ認知構造を持つ場合——「LLMが生成し、LLMが評価する」場合——改善ループは自己強化的な確信の蓄積に変わる。LLMは「改善した」と確信しながら、正しいものを書き換え続ける。

## 人間とAIの協働でも、同じ問いが立つ

毎日AIと会話している人が「最近うまく使えるようになった気がする」と感じるとき、それは本当に改善しているのか。

もし評価器がないとすれば——「なんとなくいい感じ」という主観的な印象しかなければ——それはAlphaEvolveではなく、LLM wiki自己進化と同じ構造になっている可能性がある。生成側（自分の会話スタイル）を、生成側が評価している。

思考パターンとAIへの指示スタイルを測定することは、この意味で「評価器を持つこと」に相当する。

thought-analyzerが会話ログから思考パターンを定量的に測るのは、「改善したかどうか」を確認できる基準点を作るためだ。先月と今月で `integrative_complexity` スコアがどう変わったか。`specification_precision` は上がっているか。これが評価器だ。

評価器なしの自己進化は、確信を持ちながら正しいものを書き換え続ける。
評価器ありの自己進化は、実際の変化を追跡できる。

どちらの「自己進化」をしているかは、測ってみるまで分からない。
