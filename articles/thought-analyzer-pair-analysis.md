:::message
AIとユーザーの対話ループを分析対象として設計したい開発者・研究者向け
:::

thought-analyzerはこれまで、ユーザーの発言だけを分析対象としてきた。会話ログのうち「あなたが書いたもの」だけを読み、9軸で思考パターンを抽出する。AIの返答は読まない。

なぜそう設計したか。プライバシーと測定の純粋性だ。AIの返答を含めると、「ユーザーの思考パターン」なのか「このAIの出力スタイルへの適応」なのかが判別できなくなる。

しかし、この設計には盲点があった。

**AIとの対話において、「どう指示するか」と同じくらい重要なのが、「AIの出力にどう反応するか」だ。**

その反応もまた、思考の一部だ。

---

## 何が足りなかったか

現在のthought-analyzerが測っているのは「入力側」の思考パターンだ。

- どう問いを立てるか（`need_for_cognition`）
- どう構造を組み立てるか（`integrative_complexity`）
- どう他者を想定するか（`perspective_taking`）

これは「AIに何を投げるか」の解像度を上げる。しかし「AIが返したものをどう扱うか」は測っていない。

同じ`need_for_cognition: high`のユーザーでも、AIの出力をほぼ全量採用する人と、細部を精密に修正し続ける人と、ほとんど却下して自分で書き直す人がいる。この差は、既存の9軸からは読み取れない。コーディング指示力6軸も同様だ。6軸は「指示を出す瞬間」を測るが、「出力を受け取った後」は測定対象外だ。

---

## ペア分析という発想

分析の単位を変える。

```
【従来】ユーザー発言 → 分析
【ペア分析】(AI出力 → ユーザー反応) のペア → 分析
```

AIが何かを返した。それに対してユーザーはどう反応したか。この「対話の一周」をひとつの観測単位として扱う。

ユーザーの反応は4種類に分類できる：

| 反応 | 定義 |
|------|------|
| `adopt` | AI出力をそのまま、または軽微な変更で使用 |
| `modify` | 方向性は保ちつつ、内容・表現を修正 |
| `reject` | 却下し、別の指示・別案を提示 |
| `ignore` | 出力に対するリアクションなく次の話題へ |

これを会話ログ全体で集計すると、**その人がAIの出力をどう処理するかの構造**が見えてくる。

---

## `reaction_patterns`フィールドの設計

ペア分析の出力として、新たに`reaction_patterns`フィールドを定義する。

### `reaction_distribution`

反応4種の分布をスコアとして記録する。

```json
"reaction_distribution": {
  "adopt": 0.4,
  "modify": 0.35,
  "reject": 0.15,
  "ignore": 0.1
}
```

この分布が、ユーザーがAIとどういう関係を結んでいるかの「地図」になる。adopt優位はAI委任型、modify優位は思考の精緻化に使う型、reject優位は壁打ち・検証型、ignore優位は対話の目的が別の場所にある可能性がある。

### `delegation_boundary`

何をAIに委任し、何を自分で保持するかの境界を記録する。

Risko & Gilbert（2016）の認知的オフローディング研究によれば、人は課題の性質によって認知負荷を外部に転嫁する傾向がある。AIとの対話では、「コードはAIに任せて設計は自分でやる」「要約は任せて判断は自分でやる」のように、委任の境界が個人ごとに異なる。

```json
"delegation_boundary": {
  "delegates": ["code_generation", "summarization"],
  "retains": ["architecture_decisions", "tone_judgment"]
}
```

固有名詞・技術名は含めず、抽象的なカテゴリラベルで記録する。

### `correction_precision`

修正するとき、どの精度で介入するかを測る。

```json
"correction_precision": "surgical"
// "surgical" | "directional" | "holistic"
```

- `surgical`：特定の語・文・ロジックを置き換える精密な修正
- `directional`：「この方向性で」「もっと〜に」という大まかな再指示
- `holistic`：出力全体を破棄し、ゼロから構築し直す

Hattie & Timperley（2007）のフィードバック研究では、修正の粒度が学習の質に影響することが示されている。AIとの協働においても、どの粒度で介入するかは思考スタイルの表れだ。

### `blind_spot_patterns`

AIの出力のうち、ユーザーが一貫して反応しない要素を記録する。

```json
"blind_spot_patterns": ["alternatives", "caveats"]
```

AIが複数の選択肢を提示してもユーザーが常に最初の案だけを採用する、免責・注意書き部分をスキップする、AIからの問い返しを無視する——これらの一貫したパターンは、その人の「認知的な死角」として機能している。

### `preferred_ai_style`：ズレが映す、理想の対話形式

`blind_spot_patterns`をもう一段掘り下げると、別の問いが浮かぶ。**なぜそのパターンが生じているのか**。

AIが毎回「選択肢A・B・Cがあります」と提案する。ユーザーは常にAだけを即採用してBCを無視する。これは「選択肢を求めていない」サインだ。最善案を一つ出してほしいのに、毎回選ばされている。会話は進んでいるように見えて、ユーザーにとって摩擦がある。

AIが「〜についてはどうしますか？」と問い返すたびにユーザーが無視して次の指示を出す。確認より前進を求めている。判断をAIに戻されるたびに、対話の流れが止まっている。

このズレの積み重ねから、**そのユーザーが快適に前進できるAIの出力スタイル**（`preferred_ai_style`）が推定できる。

```json
"preferred_ai_style": "decisive"
// "decisive" | "exploratory" | "concise" | "structured"
```

| 値 | 定義 | 推定されるズレのサイン |
|---|---|---|
| `decisive` | 最善案を一つ。選択肢より断言 | `blind_spot_patterns: ["alternatives"]`が一貫して出る |
| `exploratory` | 可能性を広げる。問いを深める | `modify`率が高く、常に別の角度を求めている |
| `concise` | 短く直接的に。詳細より結論 | AI出力の後半を一貫してスキップ、または「短くして」が繰り返す |
| `structured` | 段階的に整理。一気より手順 | `adopt`率が高く、構造化出力に安定して反応する |

このフィールドは9軸の`need_for_cognition`と相関する。NFC高いユーザーは選択肢より開かれた問いを好む傾向があり、AIが毎回選択肢を提示すると`blind_spot_patterns: ["alternatives"]`として行動ログに現れる。ペア分析はこの関係を自己申告なしに検出できる。

`ignore`パターンの解釈にも`preferred_ai_style`は効く。ignoreは「関係ない話題へ移った」とも読めるが、9割は「そのAI出力のスタイルに本人が納得していない」サインだ。沈黙・保留が本人の不納得から生じるのと同じ構造で、対話ログの中でそれが`ignore`として現れる。

## 既存の9軸・6軸との関係

`reaction_patterns`は既存の軸を置き換えない。測定タイミングが根本的に異なる。

| 測定領域 | タイミング | 何を測るか |
|----------|-----------|------------|
| 思考パターン9軸 | ユーザー発言の中 | どう考えるか |
| コーディング指示力6軸 | AIへの指示の中 | どう指示するか |
| `reaction_patterns` | AI出力への反応の中 | どう処理するか |

3つが揃うことで、AIとの対話サイクル全体が測定対象になる。

```
ユーザー → [9軸 + 6軸で測る] → AI → [reaction_patternsで測る] → ユーザー → ...
```

軸間に相関はあり得る。`need_for_cognition: high`のユーザーは`correction_precision: surgical`になりやすい、という仮説は検証できる。しかし相関は「同じものを測っている」とは意味が異なる。入口と出口が重なっても、通過点は別物だ。

## この設計が「処方」につながる理由

ペア分析は、thought-analyzerが「測る」だけでなく「使う」ツールになるための基盤だ。

たとえばこのような`reaction_patterns`を持つユーザーがいるとする：

```json
{
  "reaction_distribution": { "adopt": 0.6, "modify": 0.3, "reject": 0.05, "ignore": 0.05 },
  "delegation_boundary": { "retains": ["architecture_decisions"] },
  "correction_precision": "directional",
  "blind_spot_patterns": ["alternatives"],
  "preferred_ai_style": "decisive"
}
```

9軸・6軸の結果だけでは導けないが、ペア分析があればこう処方できる：

> 「AIの出力をよく採用するスタイルですが、構造的な判断は常に自分で保持しています。また、AIが複数案を提示してもほぼ最初の案を選んでいます——選択肢を必要としていない可能性があります。指示に『最善案を一つだけ出してください』と加えると、選ぶコストが下がります。構造的な制約は冒頭に明記するとさらに効果的です。」

「AIの使い方を変える」ではなく、「あなたに合った対話の形に整える」という提案だ。思考パターンの測定から、AIとの対話そのものの設計改善へ。ペア分析はその接続点だ。

## 実装上の制約と設計の誠実さ

現状のthought-analyzerはユーザー発言のみを分析する。ペア分析を実装するには、AIの返答もログに含める必要がある。これはプライバシー設計の変更を伴う。

基本方針は変えない：

- AIの返答をサーバーに送信しない
- ペア分析はローカルのみで完結させる
- 保存・送信されるのはフィンガープリントJSONのみ

「AIの返答を読む」ことと「AIの返答を保存・送信する」ことは別の問題だ。前者は分析に必要で、後者はしない。この区別を設計に明示することが、思考パターン分析ツールとして誠実な立場だと考えている。

ペア分析は現在、設計フェーズにある。実装の詳細はskill.mdの次版に追記される。

## まとめ：測定対象の拡張が問いを変える

thought-analyzerはこれまで「あなたはどう考えるか」を測ってきた。

ペア分析は問いを変える：**「あなたはAIとどう対話しているか」**。

どう指示するかと、何を受け取りどう処理するか。この2つが揃って初めて、AIとの協働における自分のスタイルが完全に見えてくる。
