---
title: "コーディング指示力7モードを読む ── 各モードの行動パターン"
---

コーディング指示力の分析が出力する7モードは、思考パターンの8類型と異なる性格を持つ。

8類型は比較的安定した認知スタイルを表す。しかし7モードは**状況によって変わる行動パターン**だ。同じ人でも慣れた領域ではSPECIFIER、未知の領域ではINQUIRERになる。「型」ではなく「モード」と呼ぶのはそのためだ。

本章では、各モードが実際の指示行動としてどう現れるかを解説する。

---

## SPECIFIER（設計指示モード）

**判定条件**：`specification_precision ≥ 4` かつ `system_abstraction ∈ {architecture, component}`

要件・制約・非機能要件まで構造的に定義してから指示を出す。AIが推測しなくて済む状態を事前に作る。

### 指示の特徴

「何を作るか」だけでなく「何を作らないか」「どの境界まで含むか」を明示する。エッジケースや依存関係を事前に整理し、システム全体の構造を把握したうえで指示する。

```
例：「この関数はユーザー認証済みの場合のみ動作する。
未認証の場合は例外を返す（リダイレクトはしない）。
外部APIとの通信はこのモジュール内に限定する。」
```

### 強みが出る場面

要件が明確で、実装の精度が重要な場面。大規模なシステム改修、チームでの仕様共有。

### 注意点

探索段階や要件が流動的な場面では、指示の精度を求めすぎることが逆にボトルネックになることがある。「完璧な仕様を書いてから動く」より「動かしながら仕様を固める」が適切な場面もある。

---

## DEBUGGER（構造診断モード）

**判定条件**：`error_recognition = structural` かつ `specification_precision ≥ 3`

AIの出力に問題が生じたとき、表面的な症状ではなく構造的な原因を特定して修正指示を出す。

### 指示の特徴

「エラーが出た」ではなく「このロジックの前提が崩れているため、呼び出し順に問題がある」という形で問題を記述できる。原因の仮説を持って修正依頼を出すため、AIが推測で動く必要がない。

```
例：「このエラーはAPIのレスポンス形式が変わったことで発生している。
パース処理の部分（L.42〜58）を修正する必要がある。
nullチェックも追加して。」
```

### 強みが出る場面

複雑なバグの修正、レガシーコードの改修、依存関係が絡んだ問題の解決。

### 注意点

構造診断に時間をかけすぎると、「まず動かしてみる」という速度を犠牲にすることがある。

---

## STRATEGIST（戦略委任モード）

**判定条件**：`decision_quality = adaptive` かつ `iteration_style ∈ {batch_clear, incremental_clear}`

全体方針を決めてAIに委ねる。状況に応じて枠組みごと変えられる（adaptive）判断力を持ち、フィードバックは明確に出す。

### 指示の特徴

AIをツールではなくチームメンバーのように扱う。「ここまでの方針でこのアウトプットを出して」という形で指示し、結果を評価して次の方針を決める。状況が変わったとき、アプローチ自体を切り替える判断が速い。

```
例：「今回はパフォーマンスより可読性を優先する方針で実装して。
完成したら、後でリファクタリングできる箇所をリストアップして。」
```

### 強みが出る場面

長期的なプロジェクト、方針が変わりやすい探索フェーズ、AIとの分業が明確な作業。

---

## ITERATOR（反復精錬モード）

**判定条件**：`iteration_style = incremental_clear`（上記以外）

小さく・明確なフィードバックを積み重ねて完成度を上げる。一度に大きな変更を依頼せず、確認しながら少しずつ進む。

### 指示の特徴

AIとの対話サイクルが細かい。一回の指示で大きな変更を求めず、出力を確認してから次のステップに進む。「少しずつ確かめながら」というリズムで動く。

```
例：「まずHTMLの構造だけ作って。
（確認後）OKです。次にCSSを追加して。
（確認後）ここの余白だけ調整して。」
```

### 強みが出る場面

複雑な実装、品質に厳しい作業、初めて触る技術領域。

---

## PATTERN（パターン適用モード）

**判定条件**：`decision_quality = routine` かつ `specification_precision ≥ 3`

既知の解決パターンを確実に適用する。新しいアプローチより実績のある手順を再利用する。安定した実装スタイルだ。

### 指示の特徴

「この前と同じやり方で」「このライブラリのベストプラクティスに従って」という形の指示が多い。新しいアプローチを試すより、チームの標準や既存のパターンに合わせる。

```
例：「前回のAPIエンドポイントと同じ構造でこれも作って。
エラーハンドリングのパターンも統一して。」
```

### 強みが出る場面

チームの標準に合わせる作業、定型的な実装、一貫性が重要なプロダクト。

---

## INQUIRER（対話探索モード）

**判定条件**：`technical_vocabulary ∈ {lay, mixed}` かつ `decision_quality ≠ deferred`

技術的な意図をAIとの対話の中で詰めていく。日常語や近似的な表現で始め、AIの返答を見ながら要件を明確化していく。

### 指示の特徴

「いい感じにして」から始まって、AIの出力を見てから「そうじゃなくて、こういう感じ」と修正する。事前の構造化より対話による収束を好む。このモードは探索段階や、自分の意図がまだ言語化されていない段階で自然に現れる。

```
例：「もう少しシンプルな感じにしたい。
（出力後）うーん、シンプルすぎた。
情報は全部残しつつ、見た目をすっきりさせたい。」
```

### 強みが出る場面

意図がまだ固まっていない探索フェーズ、デザイン・表現の調整、「言語化が難しい感覚」を扱う作業。

### 注意点

対話が長くなりすぎると、AIが蓄積した文脈の中で前の指示が後の出力を歪める可能性がある（コンテキスト同期の問題。第6章参照）。

---

## DELEGATOR（一任モード）

**判定条件**：上記いずれにも該当しない

技術的な判断をAIに委ねる。要件の定義より「AIが何を返すか」を見てから判断する傾向がある。

### 指示の特徴

細かい仕様を指定せず、AIの判断を信頼して進む。「よしなにやって」「いい感じにして」という形で始まり、返ってきたものを見てから評価する。

### DELEGATORは劣ったモードではない

DELEGATORが出る背景はさまざまだ。ドメインへの不慣れ、実験的な探索、あるいはAIへの信頼が高い。モードの出力だけで「指示力が低い」とは判断できない。`collaboration_profile`と合わせて文脈を読む必要がある。

DELEGATORが問題になるのは、「AIが返したものを検証する基準を持てなくなる」状態が習慣化したときだ。委ねること自体は戦略として有効だが、最終的な判断軸を持っているかどうかが鍵になる。

---

## モードは切り替わる

同じ人でも文脈によってモードは変わる：

- 慣れたドメイン → SPECIFIER / PATTERN
- 未知のドメイン → INQUIRER / DELEGATOR
- 問題が発生したとき → DEBUGGER
- 方針転換が必要なとき → STRATEGIST
- 品質を高めたいとき → ITERATOR

ログ全体を通して一つのモードに収束しているなら、それが習慣的なスタイルだ。複数のモードが混在しているなら、文脈に応じた切り替えができているサインとして読む。どちらが良いかは一概には言えない。
