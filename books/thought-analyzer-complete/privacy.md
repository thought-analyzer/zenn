## 設計の出発点

thought-analyzerは会話ログを入力とする。ログには個人情報・機密情報・固有名詞が含まれる可能性がある。この設計上のリスクを正面から扱い、透明性を持って設計した。

---

## 何を読み取り、何を読み取らないか

### 読み取るもの

**ユーザー自身の発言のみ。**

- 指示・質問・コメントなど、ユーザーが入力したテキスト
- 発言のパターン・スタイル・構造

### 読み取らないもの（skillが自動除外）

| 除外対象 | 理由 |
|---------|------|
| AIの返答・ツールの出力 | 分析対象はユーザーの思考パターン。AIの発言は不要 |
| コードブロックの内容 | 実装の詳細・APIキー等が含まれる可能性 |
| URL・固有名詞・サービス名・人名 | 個人・組織を特定できる情報を含む可能性 |
| APIキー・認証情報 | セキュリティリスク |
| 添付ファイル・画像の内容 | テキスト分析の対象外 |

---

## 出力に含まれるもの・含まれないもの

### フル出力JSON（画面表示）に含まれるもの

- 9軸または6軸の値（抽象化されたパターン）
- commentaryの記述（自然言語、軸名を使用）
- 理論的参照（文献リスト）

### フル出力JSONに含まれないもの

- 固有名詞・サービス名・人名
- コードの内容
- URLや技術的な実装詳細
- 会話の内容そのもの（引用・要約を含まない）

---

## データ送信の仕組み

### 送信は任意

分析結果の送信は**完全に任意**だ。送信しなくても分析結果は画面で利用できる。

### 送信されるもの

```json
{
  "schema_version": "2.0",
  "analyzed_at": "YYYY-MM",
  "message_count": 87,
  "fingerprint": {
    "abstraction_direction": "concrete_to_abstract",
    "problem_style": "fix",
    ...
  },
  "user_token": "（設定した場合のみ。元の文字列ではなくハッシュ値）"
}
```

送信されるのは**軸の値と件数のみ**。`commentary` は含まない。固有名詞・会話の内容は含まない。

### 送信されないもの

- 会話の内容・引用
- 固有名詞・サービス名
- `commentary`（strengths・blind_spotsなどの評価的記述）
- 元のログテキスト

### サイズ制限

送信JSONは2000バイト以内に設計されている。これにより、軸の値以外の情報が物理的に含まれないことを保証する。

## トークンのプライバシー設計

時系列追跡トークンは、以下のプロセスで処理される：

1. ユーザーが任意の文字列を入力
2. skillがその文字列をハッシュ化（SHA-256等）
3. **ハッシュ値のみ**をJSONに含めてサーバーに送信
4. 元の文字列はどこにも保存されない

ハッシュ値から元の文字列を逆算することは計算上不可能だ。ただし、同じ文字列を入力すれば同じハッシュ値が生成されるため、時系列照合が可能になる。

トークンを忘れた場合、過去のデータとの照合はできなくなる。新しいトークンを設定することで、その時点からの時系列追跡を再開できる。

## プライバシーに関する判断をユーザーに委ねる設計

thought-analyzerの設計原則は**「判断をユーザーに委ねる」**だ。

- 送信するかしないかはユーザーが決める
- トークンを設定するかしないかはユーザーが決める
- ログを貼り付けるかどうかはユーザーが決める

skillは分析を実行し、結果を提示する。それをどう扱うかはユーザーの判断に任される。起動時のメッセージで「読み取るもの・読み取らないもの」を明示するのも、この設計原則に基づいている。
