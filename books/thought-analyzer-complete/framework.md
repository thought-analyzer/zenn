title: "3層フレームワークの深掘り ── 発想力・構想力・指示力"
---

## なぜ3層に分けるのか

「AIをうまく使えない」という状態には、原因が3つある。発想が言語化されていないのか、構造化が不十分なのか、指示の表現が不正確なのか。これらは症状が似ていても、改善アプローチがまったく異なる。

一体として「AIを使う力」と捉えると、どこで詰まっているかが分からない。3層に分けることで、課題の所在が見えてくる。

```
発想力（あなたの内側） → 構想力（設計・整理） → 指示力（AIへの伝達）
         ↑                      ↑                      ↑
    AIには測れない          9軸で可視化             6軸で可視化
```

## 発想力：測れないものを正確に定義する

発想力はthought-analyzerが測らない唯一の層だ。その理由を正確に述べる。

**会話ログから測れないのは「発想の質」ではなく「発想の独自性」だ。**

何を指示したか（内容）は観測できる。しかしその指示の背後にある発想が「独自のものか、一般的なものか」は、比較基盤なしには評価できない。発想力の評価には大規模な比較データが必要であり、現時点のthought-analyzerはその基盤を構築中だ。

### 発想の種類

Margaret Boden（1990）は創造性を3種類に分類した：

| 種類 | 定義 | 実際の頻度 |
|------|------|-----------|
| **Combinational** | 既存アイデアの新しい組み合わせ | 最多 |
| **Exploratory** | 既存の概念空間の限界を押し広げる | 多い |
| **Transformational** | 概念空間そのものを変える | 稀 |

イノベーションの大部分はCombinationalだ。「完全に新しいひらめき」より「既知の2領域を結びつける力」の方が、実際の発想の大半を占める。

Arthur Koestler（1964）はこれをBisociation（2つの無関係なフレームの衝突）と呼んだ。異分野の知識を持ち、それを接続する思考スタイルを持つ人が、発想力が高い。

**つまり発想力を支えるのは：**
```
発想力 ≈ 知識の幅 × 経験の幅 × 結びつける思考スタイル
         （感情・動機は発想を起動させる燃料）
```

`concept_distance`（軸5）は発想力の土台となる「概念を遠い領域で結びつける思考スタイル」を間接的に示す。発想力そのものではないが、発想が生まれやすい土壌かどうかを測る。

## 構想力：9軸が測る層

構想力は「発想を実行可能な形に変換する力」だ。収集・整理・構造化の3プロセスで機能する。

| プロセス | 定義 | 対応する軸 |
|---------|------|-----------|
| **収集** | 問題に関係する情報・制約・前提を集める | `need_for_cognition`（軸7） |
| **整理** | 優先順位をつけ、不要な要素を落とす | `integrative_complexity`（軸8） |
| **構造化** | 依存関係と全体像を把握する | `abstraction_direction`（軸1）, `system_abstraction`（6軸・軸3） |

構想力が低い状態では、発想がどれだけ豊かでも「なんとなくいい感じに」という指示になる。発想の解像度は高いのに、構造化の解像度が低い。

### 9軸と構想力の対応

9軸はすべて構想力に対応するわけではない。思考スタイル全般を測る軸もある。

| 軸 | 主な対応 |
|----|---------|
| `abstraction_direction` | 構想の出発点（概念から入るか具体から入るか） |
| `problem_style` | 構想の修正スタイル |
| `perspective_taking` | 構想に他者視点が入るか |
| `face_strategy` | 構想の伝達スタイル |
| `concept_distance` | 発想の土壌（構想力の前段階） |
| `evaluation_framing` | 構想を評価・提示する枠組み |
| `need_for_cognition` | 構想の深さへの欲求 |
| `integrative_complexity` | 構想の構造的複雑さ |
| `epistemic_curiosity` | 構想を広げる動機 |

## 指示力：6軸が測る層

指示力は「構想をAIに正確に伝え、実現させる力」だ。技術的なスキルではなく、思考のパターンとして現れる。

| 6軸 | 指示力の何を測るか |
|-----|-----------------|
| `specification_precision` | 構想を要件に変換できるか |
| `error_recognition` | AIの出力のどこを見て問題を判断するか |
| `system_abstraction` | システム全体の文脈で指示できるか |
| `decision_quality` | 技術選択・スコープ判断を自分でできるか |
| `technical_vocabulary` | 正確な語彙で意図を伝えられるか |
| `iteration_style` | フィードバックの粒度と明確さ |

## 2つのskillを組み合わせて読む

9軸と6軸は独立した分析だが、組み合わせることで「一貫性」と「文脈依存」が見えてくる。

**一貫性の例：**
- 9軸で `problem_style: fix`（修正で対応する傾向）
- 6軸で `decision_quality: routine`（既知パターンを適用）
→ 両方が「慎重・適応型」を示している場合、それはスタイルとして安定している

**文脈依存の例：**
- 9軸で `problem_style: pivot`（枠を変える傾向）
- 6軸で `decision_quality: deferred`（技術判断は委ねる）
→ 一般的思考では発散するが、技術文脈では判断を委ねる。コーディング文脈での経験不足か、または信頼するパートナーへの意図的な委任の可能性
